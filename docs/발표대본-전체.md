## 주제 1: GitLab CI/CD 기반 사내망 자동 배포 시도 결과

---

### 1. 시도 목적 / 목표 구조 (2분)

이번 시도의 목적은 **GitLab CI/CD를 이용해서 사내망 개발 환경, 구체적으로는 제 PWC에 자동 배포 환경을 구축하는 것**이었습니다.

의도했던 구조를 설명드리면, GitLab의 리포지토리에서 코드를 푸시하면, GitLab CI Pipeline이 동작하고, 공용 Runner인 shared-kr-runner가 HCloud에서 실행됩니다. 그 Runner가 Docker 이미지를 띄워서 빌드와 배포 스크립트를 실행하고, 마지막으로 SSH로 제 PWC, 즉 10.90.170.168에 배포하는 구조였습니다.

이런 구조는 이론적으로는 완전히 가능하고, 다른 환경에서는 실제로 많이 사용되는 방식입니다.

---

### 2. 진행한 작업 (2분)

실제로 진행한 작업을 두 부분으로 나눠서 말씀드리겠습니다.

**첫 번째, CI/CD 구성은 완료했습니다.** .gitlab-ci.yml 파일과 deploy.sh 같은 배포 스크립트를 모두 작성했고, GitLab Variables에 필요한 키와 환경변수를 설정했습니다. 공용 Runner 사용 설정도 완료했습니다. 이 부분은 문제가 없었습니다.

**두 번째, 네트워크와 보안 절차도 완료했습니다.** SRMS를 통해서 방화벽 신청을 했는데, 출발지는 10.11.95.229, 즉 HCloud의 공용 Runner이고, 목적지는 제 PWC인 10.90.170.168의 22번 포트입니다. 이 신청이 승인되었습니다.

다만 특이사항이 있었는데, 담당 팀에서 **"테스트 목적으로 6개월만 한시적으로 허용한다"**는 안내를 받았습니다. 그 이후에는 사용할 수 없다고 하더라고요.

여기까지는 다 잘 진행됐습니다. 문제는 그 다음부터였습니다.

---

### 3. 핵심 장애 (2분)

핵심적으로 막힌 지점은 **PWC에서 SSH 수신 환경을 구성할 수 없었다**는 것입니다.

제 PWC는 Windows PC입니다. 개인 단말, 팀 단말 성격이죠. 그런데 사내 보안 정책과 권한 정책 때문에, Defender나 백신, 로컬 관리자 권한 제한 등으로 인해서 **OpenSSH를 설치하고 활성화하는 시도 자체가 어려웠습니다.**

구체적으로 확인해보니, `netstat -an | findstr :22` 명령어로 확인했을 때, **22번 포트를 LISTEN 상태로 만들 수 없었습니다.**

이게 무슨 뜻이냐면, 외부에서 SSH로 접속하려고 해도 받아줄 서비스가 없다는 겁니다. 즉, 외부에서 SSH로 배포를 "받는" 구조 자체가 성립하지 않는 거죠.

Runner는 배포를 시도하는데, PWC는 받을 준비가 안 되어 있는 상태였습니다.

---

### 4. 왜 방화벽을 열어도 안 됐나 (1분)

여기서 의문이 생길 수 있습니다. "방화벽은 승인받았는데 왜 안 되는 거지?"

**한 문장으로 정리하면, 방화벽은 열렸지만, 목적지인 PWC에 SSH 서비스 자체를 올릴 수 없어서 접속이 불가능했습니다.**

비유를 들자면, 네트워크는 길이고, 엔드포인트는 문입니다. SRMS 방화벽 승인으로 길은 열렸습니다. 정상입니다. 그런데 PWC의 정책과 권한 제약 때문에 문을 열 수 없었습니다. 차단된 거죠.

**길은 열렸지만, 문을 열 수 없는 상황**이었습니다.

이건 네트워크 문제라기보다는, 엔드포인트의 정책 문제였던 겁니다.

---

### 5. 구조적 한계 & 결론 (2분)

이 과정에서 구조적인 한계를 발견했습니다.

**첫째, PWC나 팀 PC는 자동 배포 대상 서버의 요건을 충족하지 못합니다.** 서버는 상시 서비스가 구동되어야 하고, 접근 통제가 되어야 하며, 운영 관리가 되어야 합니다. 그런데 Windows PC는 이런 요건을 충족하지 못합니다. 개인 단말 성격이기 때문에 보안 정책도 훨씬 엄격하죠.

**둘째, 외부 공용 Runner에서 사내 단말로 직접 접근하는 구조는 정책적으로 제약이 큽니다.** 공용 Runner는 HCloud 환경, 즉 10.11.x.x 대역인데, 사내망 10.90.x.x로의 접근은 원칙적으로 제한되어 있습니다. 이번에 테스트 용도로 6개월만 허용받긴 했지만, 장기적인 해결책이 아닙니다.

**최종 결론은, 현재 환경인 PWC나 팀 PC에서는 사내망 CI/CD 자동 배포가 불가능하다는 것입니다.**

이유를 정리하면, PWC에서 SSH 서버를 설치하고 활성화하는 시도 자체가 정책상 어렵고, 공용 Runner에서 사내 단말로의 접근 구조가 정책적으로 제약이 크며, 개인이나 팀 PC는 CI/CD 배포 대상 서버 요건을 충족하지 못합니다.

---

### 6. 대안 & 느낀점 (1분)

그러면 향후 어떻게 해야 할까요?

**정식 프로젝트화가 필요합니다.**

먼저 GMDM에 프로젝트를 생성해야 합니다. AI TFT 과제 같은 형태로요. 그리고 승인된 서버나 VM을 할당받아야 합니다. 이건 운영팀 관리 하에 SSH와 방화벽 설정이 가능한 환경이어야 합니다. Linux 기반 서버 환경이 이상적이겠죠.

또한 **GitHub Self-hosted Runner를 활용**하는 방법도 있습니다. 사내망 내부에 Runner를 직접 구축하면, 외부에서 사내로 접근하는 구조가 아니라 내부에서 내부로 배포하는 구조가 되기 때문에 훨씬 자연스럽습니다.

**느낀점을 솔직하게 말씀드리면, 결론적으로 사내망에서 배포하기가 너무 힘들었습니다.** 설정 문제보다 정책, 권한, 보안 제약이 더 큰 허들이었고요. 다음에는 정식 서버와 프로젝트 선행이 필수라고 느꼈습니다.

한마디로, 다 막아놨더라고요. (웃음)

**교훈은, CI/CD 구성 자체는 완료했으나, 배포 대상 환경 확보가 선행되어야 한다는 것입니다.** 정식 프로젝트로 다시 시도할 예정입니다.

이상으로 GitLab CI/CD 시도 결과 공유를 마치겠습니다.

## 주제 2: Vibe Coding 교재 기반 실습 사례

---

### 오프닝 (30초)

주제 2는 "Vibe Coding으로 완성한 MY-TODOLIST 프로젝트"입니다.

여러분은 혹시 "Vibe Coding"이라는 단어를 들어보셨나요? AI 전문가 Andrej Karpathy가 처음 제시한 개념으로, AI와 함께하는 새로운 개발 패러다임입니다.

저는 Vibe Coding 교재를 기반으로 이 방식을 실제로 적용해서 단 4일 만에 풀스택 웹 애플리케이션을 완성했습니다. 오늘은 교재의 순서를 따라가면서, 각 단계에서 어떻게 Vibe Coding을 적용했는지 공유하겠습니다.

---

### 1. Vibe Coding 개요 (5분)

#### Vibe Coding이란?

먼저 Vibe Coding이 무엇인지 설명드리겠습니다.

Vibe Coding은 AI 전문가 Andrej Karpathy가 처음 제시한 개념입니다. **개발자가 생성형 인공지능의 도움을 받아 자연어 프롬프트를 통해 코드를 작성하는 프로그래밍 방식**입니다.

어떻게 작동하냐면, 개발자가 Project나 Task를 LLM에 설명하면, LLM이 프롬프트를 기반으로 코드를 생성합니다. 그러면 개발자는 그 결과를 평가하고 LLM에 개선사항을 요청하는 방식으로 개발이 진행됩니다. 핵심은 코드를 세세하게 관리하지 않고, **반복적인 실험에 집중**한다는 것입니다.

#### Vibe Coding의 특징

Vibe Coding에는 세 가지 주요 특징이 있습니다.

첫째, **자연어 기반 개발**입니다. 영어든 한국어든 모국어로 작성한 프롬프트만 입력하면 원하는 결과를 얻을 수 있습니다.

둘째, **AI 의존적 코딩**입니다. 코드의 작동 원리를 완전히 이해하지 않아도 코드를 작성할 수 있습니다. 이게 논란이 되기도 하지만, 빠른 프로토타이핑에는 매우 유용합니다.

셋째, **직감적 접근**입니다. 사전에 엄밀한 논리나 설계를 바탕으로 하지 않고, 직감과 느낌에 의존해서 프로그래밍을 진행합니다.

#### 기존 코딩 vs Vibe Coding

화면의 표를 보시면 기존 코딩과 Vibe Coding의 차이를 한눈에 비교할 수 있습니다.

**개발자의 역할**이 달라집니다. 기존에는 설계자, 개발자, 디버거를 모두 담당했다면, Vibe Coding에서는 프롬프트 작성, 가이드, 테스터 역할을 합니다.

**코드 생성** 방식도 다릅니다. 기존에는 개발자가 직접 작성했지만, Vibe Coding에서는 AI가 자동으로 생성합니다.

**전문 지식** 요구 수준도 낮아집니다. 기존에는 높은 프로그래밍 언어 지식이 필요했지만, Vibe Coding에서는 상대적으로 낮은 기술적 지식만으로도 개발이 가능합니다.

**개발 속도**는 프로토타입 단계에서 훨씬 빠릅니다. 기존 방식은 체계적이지만 느린 반면, Vibe Coding은 빠른 반복이 가능합니다.

**디버깅** 방식도 변화합니다. 기존에는 수동으로 코드를 이해해야 했지만, Vibe Coding에서는 AI 피드백 기반으로 반복적으로 자동 수정합니다.

#### VDD (Vibe-Driven Development)

교재에서 소개하는 중요한 개념 중 하나가 바로 **VDD**, Vibe-Driven Development입니다.

전통적인 SDLC는 계획, 설계, 구현, 테스트, 배포 순으로 진행됩니다. 반면 VDD는 아이디어에서 시작해서 **프롬프트 기반 설계**, **프롬프트 기반 구현**, 테스트, 그리고 **개선 반복**이라는 사이클을 빠르게 돕니다.

핵심은 빠른 피드백 루프와 점진적 개선입니다. 한 번에 완벽한 설계를 하는 것이 아니라, 빠르게 만들고, 테스트하고, 개선하는 것을 반복합니다.

#### TDD vs VDT

테스트 방법론도 달라집니다. 기존의 **TDD**는 Test-Driven Development로, Red → Green → Refactor 사이클을 따릅니다. 테스트 코드를 먼저 작성하고, 모든 요구사항을 안정적으로 검증하는 것이 목표입니다.

반면 **VDT**, Vibe-Driven Testing은 **Vibe → Test → Refine** 사이클을 따릅니다. 현재 개발 흐름과 AI 피드백을 활용해서 핵심 기능과 위험 지점을 신속하게 검증하는 것이 목표입니다.

실제로 저는 이 프로젝트에서 VDT 방식으로 119개의 테스트를 작성했고, 커버리지는 94~95%를 달성했습니다.

#### AI-DLC

또 하나 중요한 개념이 **AI-DLC**, AI-Driven Life Cycle입니다.

이것은 AI를 SDLC의 '중심 동료'로 두고, 모든 개발 단계에 AI를 일관되게 활용하는 방법론입니다. 요구사항 분석부터 설계, 구현, 테스트까지 모든 단계에서 AI를 활용합니다.

목표는 속도와 품질을 동시에 끌어올리는 것인데, 교재에서는 수십 배까지도 가능하다고 합니다.

#### CI/CD와 Vibe Coding

마지막으로, Vibe Coding은 CI/CD와 궁합이 매우 좋습니다.

왜냐하면, Vibe Coding은 작은 단위의 변경이 많고, 계속 바뀌는 요구사항에 빠르게 대응하며, AI가 주도적으로 테스트를 작성하기 때문입니다. 또한 운영 피드백을 빠르게 반영할 수 있습니다.

실제로 저는 GitHub Actions와 Vercel을 활용해서 완전 자동화된 배포 파이프라인을 구축했습니다.

### 2. 환경 설정 (2분)

Vibe Coding을 시작하려면 적절한 개발 환경이 필요합니다.

필수 도구로는 Node.js 18 이상, PostgreSQL 17, Git과 GitHub, 그리고 VS Code나 Cursor AI가 있습니다. 가장 중요한 것은 **Claude Code CLI**입니다.

Claude Code는 터미널 기반의 AI 코딩 어시스턴트입니다. npm으로 간단하게 설치할 수 있습니다.

설치 후에는 MCP, 즉 Model Context Protocol 설정이 필요합니다. `.mcp.json` 파일로 필요한 서버들을 구성하는데, 예를 들어 PostgreSQL과 연동하려면 postgresql-mcp 서버를 설정합니다.

프로젝트를 시작할 때는 먼저 GitHub 리포지토리를 생성합니다. `gh repo create` 명령으로 리포지토리를 만들고, 클론한 후 VS Code나 Cursor로 엽니다.

이렇게 기본 환경 설정이 완료되면 본격적인 개발을 시작할 수 있습니다.

### 3. 프롬프트 엔지니어링 (3분)

Vibe Coding에서 가장 중요한 것은 바로 프롬프트입니다. 명확하고 구체적인 프롬프트를 작성해야 정확한 결과를 얻을 수 있습니다.

좋은 프롬프트의 조건은 네 가지입니다. 명확한 컨텍스트를 제공하고, 구체적인 요구사항을 전달하며, 제약조건을 명시하고, 예상 결과를 설명하는 것입니다.

교재에서 강조하는 것은 **계층적 컨텍스트 관리**입니다.

가장 상위에는 CLAUDE.md라는 전역 설정이 있고, 그 아래 프로젝트 메모리(MEMORY.md), 도메인 정의서, PRD(요구사항), 설계 원칙, 실행 계획이 순차적으로 연결됩니다.

이렇게 계층적으로 구성하면 AI가 프로젝트의 전체 컨텍스트를 이해하고 일관된 코드를 생성할 수 있습니다. 각 계층이 다음 계층의 기반이 되는 것이죠.

CLAUDE.md는 AI에게 주는 지침서입니다. 전역 지침에는 프로젝트 전반에 적용되는 원칙을 작성합니다. 예를 들어, "간단한 코드는 주석을 지정하지 않고, 복잡한 코드일 때만 주석을 부여한다"는 식입니다.

또한 "오버엔지니어링 금지"와 "지침에 있는 기능만 정확하게 구현"이라는 제약을 명시해서, AI가 창의적으로 불필요한 작업을 하지 않도록 제어합니다.

백엔드나 프론트엔드 같은 하위 디렉토리에는 해당 영역에 특화된 CLAUDE.md를 작성합니다. 예를 들어 백엔드 CLAUDE.md에는 "SOLID 원칙 준수", "Clean 아키텍처 준수"같은 아키텍처 원칙과, "Node.js + Express + TypeScript 사용", "PostgreSQL은 pg 드라이버 사용, Prisma 금지" 같은 기술 스택을 명시합니다.

이렇게 하면 AI가 백엔드 개발을 할 때 이 지침을 따르게 됩니다.

---

### 4. Vibe Coding 도구 (3분)

Vibe Coding의 핵심 도구는 Claude Code CLI입니다. 터미널 기반의 AI 코딩 어시스턴트로, 파일을 읽고 쓸 수 있고, 코드를 생성하거나 수정할 수 있으며, Git과 통합되어 있어서 커밋이나 PR 생성도 자동으로 할 수 있습니다.

MCP는 Model Context Protocol의 약자로, AI가 외부 시스템과 통신할 수 있게 해주는 프로토콜입니다. 저는 PostgreSQL MCP로 데이터베이스와 연동하고, Supabase MCP로 프로덕션 DB와 연동했으며, Playwright MCP로 E2E 테스트를 자동화하고, Chrome DevTools MCP로 브라우저 디버깅을 자동화했습니다.

교재에서 강조하는 중요한 개념이 **서브에이전트**입니다. 서브에이전트는 특정 역할에 특화된 AI 에이전트입니다. Plan Agent는 구현 계획을 수립하고, Explore Agent는 코드베이스를 탐색하며, Backend Developer는 백엔드 개발을, Frontend Developer는 프론트엔드 개발을 담당합니다.

중요한 점은 이 서브에이전트들을 **병렬로 실행**할 수 있다는 것입니다. 예를 들어 코드 분석과 이슈 확인을 동시에 진행할 수 있습니다.

가장 강력한 도구는 **사용자 정의 Command**입니다. `.claude/commands/` 디렉토리에 마크다운 파일로 작성하는데, 예를 들어 `issue-resolver-backend.md`는 GitHub Issue 번호를 받아서 자동으로 이슈를 해결하는 커맨드입니다.

이슈 확인, 브랜치 생성, 코드 분석, 계획 수립, 테스트 작성, 문제 해결, 테스트 수행까지 전체 프로세스를 자동화할 수 있습니다. 실제로 저는 이 커맨드로 백엔드와 프론트엔드의 모든 이슈를 처리했습니다.

---

### 5. 프로젝트 기획 및 설계 (4분)

교재에서는 프로젝트 시작 전에 **도메인 정의서**를 작성하는 것을 강조합니다.

도메인 정의서란 특정 업무나 제품이 다루는 문제 영역을 공통 언어로 정리해서, 범위, 개념, 규칙, 경계를 명확히 하는 기준 문서입니다. 핵심 항목으로는 비즈니스 컨텍스트, 도메인 언어, 도메인 모델, 비즈니스 규칙, 업무 프로세스가 있습니다.

도메인 정의서도 AI로 작성할 수 있습니다. 프롬프트에는 문제 정의, 주요 기능, 필수 사항을 명확하게 작성합니다.

도메인 정의서 다음에는 **PRD**를 작성합니다. 도메인 정의서는 "무엇을"과 "왜"를 명확히 하지만, PRD는 여기에 "어떻게"와 "언제"가 추가됩니다. PRD에는 타겟 사용자 페르소나, UI/UX 요구사항, 기술 스택 선택 근거, MVP 범위 정의, 성과 지표(KPI) 등이 포함됩니다.

PRD가 완성되면 실행 계획을 수립합니다. 저는 프로젝트를 3개의 마일스톤으로 나누었습니다. M1은 DB와 백엔드 API로 1일, M2는 프론트엔드로 2일, M3는 테스트와 배포로 1일을 할당했습니다. 각 마일스톤은 다시 작은 Task들로 분할됩니다.

교재에서는 각 Task를 GitHub Issue로 관리하는 것을 권장합니다. AI가 자동으로 각 Task에 대한 Issue를 생성합니다. 각 Issue에는 Title, Label, Todo 리스트, 완료 조건, 기술적 고려사항, 의존성을 모두 포함시킵니다. 이렇게 하면 개발 진행 상황을 투명하게 관리할 수 있고, 팀원들과 효과적으로 소통할 수 있습니다.

---

### 6. 백엔드 개발 (5분)

*(시간 관계상 주요 내용만 요약)*

백엔드 개발의 핵심은 **Custom Command를 활용한 자동화**입니다. `/issue-resolver-backend 23`처럼 Issue 번호만 입력하면, AI가 자동으로 Issue 확인부터 테스트 작성, 문제 해결, PR 생성까지 전체 프로세스를 진행합니다.

교재에서 특히 강조하는 것이 **로깅**입니다. 충실하게 로깅하도록 개발하고, 로그를 기반으로 분석하도록 해야 합니다. 디버깅할 때 로그를 AI에게 제공하면 훨씬 빠르게 해결할 수 있습니다.

결과적으로 총 35개의 엔드포인트를 가진 백엔드 API를 완성했습니다.

---

### 7. 프론트엔드 개발 (5분)

*(시간 관계상 주요 내용만 요약)*

프론트엔드의 흥미로운 점은 **화면 캡처를 이용한 스타일 추출**입니다. 원하는 디자인의 화면을 캡처해서 AI에게 보여주면, 자동으로 색상, 폰트, 컴포넌트 스타일을 추출해서 가이드를 만들어줍니다.

또한 **Chrome DevTools MCP**를 활용하면 브라우저의 개발자 도구를 AI가 자동으로 분석해서 문제를 찾아줍니다. 디버깅 시간을 크게 단축할 수 있습니다.

---

### 8. 배포 (5분)

*(시간 관계상 주요 내용만 요약)*

배포는 Supabase와 Vercel을 활용했습니다. 로컬 데이터베이스의 스키마를 Supabase로 마이그레이션하는 것도 AI가 자동으로 처리합니다.

Vercel for GitHub을 사용하면 CI/CD 파이프라인이 자동으로 구성됩니다. Git Push만 하면 자동으로 빌드, 테스트, 배포까지 진행됩니다.

---

### 9. 실제 개발 성과 (3분)

실제 개발 성과를 말씀드리겠습니다.

총 개발 기간은 4일이었습니다. M1 백엔드는 1일 동안 35개 엔드포인트와 35개 테스트를 작성했고, 커버리지는 94%입니다. M2 프론트엔드는 2일 동안 7개 페이지와 84개 테스트를 작성했고, 커버리지는 95%입니다. M3 배포는 1일 동안 Vercel과 Supabase에 프로덕션 배포를 완료했습니다. 총 119개의 자동화 테스트를 작성했습니다.

모든 성능 목표를 달성했습니다. API 응답 시간은 평균 40ms, 할 일 목록 조회는 평균 3.6ms밖에 걸리지 않습니다. 테스트는 119개 전부 통과했습니다.

### 10. 회고 및 배운 점 (4분)

Vibe Coding의 장점을 정리하면,

첫째, **개발 속도가 비약적으로 향상**됩니다. 4일 만에 풀스택 애플리케이션을 완성했습니다.

둘째, **품질이 향상**됩니다. 119개의 자동화 테스트, 커버리지 94~95%를 달성했습니다.

셋째, **학습 효과**가 뛰어납니다. 최신 기술 스택을 빠르게 습득할 수 있었습니다.

넷째, **VDD를 실천**할 수 있었습니다. 빠른 피드백 루프와 점진적 개선이 가능했습니다.

도전 과제도 있었습니다. 프롬프트 엔지니어링, AI 생성 코드 검토, 통합 과정, 배포 환경 설정에서 세심한 주의가 필요했습니다.

Vibe Coding을 시작하려는 분들께 권장 사항을 드리자면, 명확한 설계를 먼저 하고, 단계별로 진행하며, VDT를 적용하고, 문서화하며, 도구를 활용하고, 검토를 필수로 하시기 바랍니다.

교재를 통해 배운 핵심 교훈은, VDD는 빠른 피드백 루프를 가능하게 하고, AI-DLC는 모든 개발 단계에 AI를 일관되게 활용하며, VDT는 개발 흐름 기반 테스트를, CI/CD 통합은 Vibe Coding과 자동화의 시너지를 만들어낸다는 것입니다.

## 예상 질문 및 답변

### 주제 1 관련 질문

**Q1: SRMS 방화벽은 승인받았는데, 왜 6개월만 허용된 건가요?**

A: 담당 팀에서 개인 PC(PWC)로의 외부 접근은 보안상 일반적으로 허용하지 않는데, 테스트 목적이라는 점을 고려해서 예외적으로 6개월만 한시적으로 허용해준 것입니다. 장기적으로는 정식 프로젝트와 승인된 서버가 필요하다는 의미였습니다.

**Q2: OpenSSH를 PowerShell이나 관리자 권한으로 설치하려고 시도했나요?**

A: 네, 시도했습니다. Windows 10/11의 선택적 기능에서 OpenSSH 서버를 설치하려고 했고, PowerShell로도 시도했지만, 사내 보안 정책(Defender, AppLocker 등)과 관리자 권한 제한으로 인해 서비스 등록과 기동 자체가 차단되었습니다.

**Q3: WSL(Windows Subsystem for Linux)를 사용하면 가능하지 않을까요?**

A: 좋은 질문입니다. WSL도 시도해봤는데, 마찬가지로 정책상 WSL 설치와 활성화에 제약이 있었습니다. 또한 WSL 내부의 SSH 서버를 Windows 호스트의 22번 포트로 포워딩하는 것도 방화벽 정책에 막혔습니다.

**Q4: GitHub Self-hosted Runner는 어떻게 구성하나요?**

A: GitHub Self-hosted Runner는 사내망 내부의 서버나 VM에 GitHub Actions Runner를 직접 설치하는 방식입니다. 이렇게 하면 외부에서 사내로 접근하는 것이 아니라, 사내 Runner가 GitHub에서 작업을 가져와서 사내 서버에 배포하는 구조가 됩니다. 하지만 이것도 정식 서버가 필요합니다.

**Q5: 다른 팀에서는 사내망 자동 배포를 어떻게 하나요?**

A: 정식 프로젝트로 등록된 경우, GMDM을 통해 서버를 할당받고, 운영팀 관리 하에 SSH, 방화벽, Runner 등을 설정합니다. 또는 HCloud에 VM을 생성해서 사용하기도 합니다. 공통점은 '정식 서버'를 사용한다는 것입니다.

### 주제 2 관련 질문

**Q6: VDD와 전통적인 애자일의 차이는 무엇인가요?**

A: 애자일도 반복적인 개발을 강조하지만, VDD는 AI를 중심에 두고 훨씬 빠른 피드백 루프를 만든다는 차이가 있습니다. 애자일은 보통 1~2주 스프린트인 반면, VDD는 하루나 심지어 몇 시간 단위로 빠른 반복이 가능합니다.

**Q7: AI가 생성한 코드의 품질을 어떻게 보장하나요?**

A: 세 가지 방법으로 품질을 보장합니다. 첫째, TypeScript의 strict 모드로 타입 안정성을 확보합니다. 둘째, 자동화 테스트를 119개 작성하여 커버리지 94~95%를 달성했습니다. 셋째, 코드 리뷰를 통해 보안 취약점과 비즈니스 로직을 검증합니다.

**Q8: 4일 개발의 비결은 무엇인가요?**

A: 세 가지입니다. 첫째, **명확한 설계**입니다. 도메인 정의서와 PRD를 먼저 작성해서 무엇을 만들지 명확히 했습니다. 둘째, **자동화**입니다. Custom Command로 Issue 해결 전체 프로세스를 자동화했고, CI/CD로 배포도 자동화했습니다. 셋째, **VDD와 VDT 적용**입니다. 빠른 피드백 루프로 개발하고, AI 기반 테스트로 품질을 보장했습니다.

**Q9: 실무에 Vibe Coding을 적용하려면 어떻게 시작해야 하나요?**

A: 작은 프로젝트나 프로토타입부터 시작하는 것을 추천합니다. 도메인 정의서와 PRD를 AI로 작성하고 검토하는 것부터 시작하세요. Custom Command를 작성해서 반복 작업을 자동화하고, 테스트도 AI로 작성하되 반드시 검토하세요. 점진적으로 적용 범위를 넓혀가는 것이 좋습니다.

**Q10: Vibe Coding의 한계는 무엇인가요?**

A: 몇 가지 한계가 있습니다. 첫째, 명확한 요구사항이 없으면 AI도 정확한 결과를 만들 수 없습니다. 둘째, 복잡한 비즈니스 로직이나 도메인 지식은 여전히 개발자가 정의해야 합니다. 셋째, AI 생성 코드는 반드시 검토가 필요합니다. 보안 취약점이나 오버엔지니어링 문제가 있을 수 있기 때문입니다. 하지만 이런 한계를 인식하고 적절히 활용하면 생산성을 크게 높일 수 있습니다.
